# RAG Evaluation Runner

## Purpose

This specification defines an end-to-end evaluation runner for the OCI RAG Agent
Blueprint.

The runner consumes a golden dataset JSONL file, invokes a running RAG agent for
each question, records the agent response and references, and uses the
evaluation model as an LLM judge to classify retrieval quality, grounding,
answer correctness, and hallucination risk.

## Scope

This specification covers:

- Command-line execution of an evaluation run against a reachable agent
  endpoint.
- Reading golden dataset records generated by
  `management.evals.generate_golden_dataset`.
- Invoking the agent `/responses` endpoint once per golden question.
- Recording agent responses, references, errors, and timing information.
- Checking whether expected source PDF and page evidence appears in references.
- Using the evaluation model configured through `EVAL_*` environment variables
  as an LLM judge.
- Writing machine-readable JSONL results and a concise summary report.
- Printing a concise console table with total, pass, review, fail, and error
  counts.
- Classification labels, judge prompt requirements, error handling, and unit
  test expectations.

This specification does not yet define:

- A web UI for evaluation results.
- Scheduled or hosted evaluation jobs.
- Integration with Langfuse datasets or external experiment trackers.
- Statistical significance analysis across repeated runs.
- Automatic remediation or document ingestion changes based on evaluation
  failures.

## Design Overview

The evaluation runner is an offline management command. It must not be part of
the FastAPI serving path.

The runner requires a running agent endpoint. The endpoint may be local, for
example:

```text
http://localhost:8080/responses
```

or a public Hosted Application invoke endpoint that exposes `/responses`.

The first implementation targets endpoints reachable without JWT authentication
to keep the evaluation loop fast. Future implementations may add optional
Bearer token support.

The runner performs this flow for each golden record:

1. Read `question`, `answer`, `source_pdf_name`, and `page_number` from the
   dataset.
2. Call the agent endpoint with a new streaming conversation, matching the
   reference UI behavior.
3. Parse the SSE stream and store the returned answer text, `references`,
   `usage`, `response_id`, and errors.
4. Run deterministic reference checks against the expected PDF and page.
5. Ask the evaluation model to judge the agent answer against the expected
   answer and retrieved references.
6. Write one JSONL result row.
7. Print and write an aggregate summary.

## Command-Line Interface

The runner must be invokable with:

```bash
python -m management.evals.run_rag_evaluation \
  --endpoint http://localhost:8080/responses \
  --dataset evals/datasets/golden_dataset.jsonl
```

Required arguments:

- `--endpoint`: full agent `/responses` endpoint URL.
- `--dataset`: golden dataset JSONL path.

Optional arguments:

- `--output`: result JSONL path. Default:
  `evals/reports/rag_eval_results.jsonl`.
- `--summary-output`: summary JSON path. Default:
  `evals/reports/rag_eval_summary.json`.
- `--max-records`: limit records for smoke testing.
- `--request-timeout-seconds`: timeout for each agent request. Default: `120`.
- `--judge-timeout-seconds`: timeout for each judge request. Default: `120`.
- `--no-progress`: disable interactive progress reporting.
- `--continue-on-error`: continue when an agent or judge call fails. Default:
  enabled.
- `--fail-on-threshold`: return non-zero when configured thresholds are not met.
  Threshold values may be added in a later implementation.

The runner must read the judge model configuration from the existing evaluation
environment variables:

- `EVAL_OCI_REGION`
- `EVAL_OCI_COMPARTMENT_ID`
- `EVAL_OCI_PROJECT_ID`
- `EVAL_OCI_MODEL_ID`
- `EVAL_GENERATION_TEMPERATURE`

The runner must share existing OCI authentication settings:

- `OCI_AUTH_MODE`
- `OPENAI_API_KEY`, when API-key mode is used.
- `OCI_CONFIG_FILE`, when config-file mode is used.
- `OCI_PROFILE`, when config-file mode is used.

The judge must not use the agent runtime model settings such as `OCI_MODEL_ID`.

## Agent Request Contract

The runner must call the agent with the same streaming contract used by the
reference Next.js UI:

```json
{
  "new_conversation": true,
  "user_request": "<golden question>",
  "stream": true
}
```

The HTTP request must send `Accept: text/event-stream` and
`Content-Type: application/json`.

The runner must parse the SSE events documented in
[Agent API Usage](../docs/agent-api-usage.md), including:

- `metadata`: conversation identifier.
- `token`: incremental answer text.
- `references`: retrieved references.
- `usage`: token usage.
- `done`: final conversation and response identifiers.
- `error`: structured stream failure.

Some hosted gateways may preserve `data:` frames but strip explicit `event:`
lines. The runner must infer event names from payload keys where possible, using
the same behavior as the reference CLI and UI parser.

If the agent returns a non-2xx HTTP status, invalid SSE payloads, or an `error`
event, the result row must preserve the failure and skip judge scoring for that
record.

## Reference Matching

For each golden record, the runner must check whether the expected source PDF
and page appear in the agent references.

Golden record fields:

```json
{
  "source_pdf_name": "source.pdf",
  "page_number": 1
}
```

Reference shapes may vary by Responses API and by the project reference
extraction logic. The reference checker must therefore normalize references
defensively.

Expected source PDF matching:

- Match case-insensitively.
- Accept exact file name matches.
- Accept references whose path or title ends with the expected file name.
- Ignore surrounding URL encoding or path prefixes when possible.

Expected page matching:

- Match numeric page fields when present.
- Accept one-based page numbers.
- Accept page number strings when present.
- If no page information is present in any reference, classify page evidence as
  `unknown`, not as a hard failure.

Reference check output:

```json
{
  "expected_pdf_found": true,
  "expected_page_found": true,
  "reference_match_status": "pdf_and_page_found"
}
```

Allowed `reference_match_status` values:

- `pdf_and_page_found`: expected PDF and page are both found.
- `pdf_found_page_missing`: expected PDF is found, but expected page is not.
- `pdf_found_page_unknown`: expected PDF is found, but references do not expose
  page information.
- `pdf_missing`: expected PDF is not found.
- `no_references`: the agent returned no references.
- `agent_error`: the agent request failed.

## LLM Judge

The runner must use the evaluation model as a judge through the
OpenAI-compatible Responses API.

The judge input must include:

- Golden question.
- Expected answer.
- Agent answer.
- Expected source PDF name and page number.
- Agent references, normalized and compacted.
- Deterministic reference check output.

The judge must be instructed to compare the agent answer to the expected answer
semantically, not by exact string matching.

The judge must return strict JSON:

```json
{
  "answer_correctness": "correct",
  "grounding": "grounded",
  "hallucination_risk": "low",
  "overall": "pass",
  "confidence": 0.0,
  "rationale": "short explanation"
}
```

The judge must not require the agent answer to match the expected answer word
for word. It must reward semantically equivalent answers, even when phrased
differently.

The judge must penalize answers that introduce unsupported facts, contradict the
expected answer, or answer a different question.

## Classification Labels

### Answer Correctness

Allowed values:

- `correct`: the answer is semantically equivalent to the expected answer and
  addresses the question.
- `partially_correct`: the answer contains the main idea but misses important
  constraints, caveats, or details.
- `incorrect`: the answer contradicts the expected answer or answers a
  different question.
- `not_answered`: the answer refuses, says it does not know, or is empty.
- `judge_error`: the judge could not produce a valid classification.

### Grounding

Allowed values:

- `grounded`: the answer is supported by references, and expected PDF evidence
  is present.
- `weakly_grounded`: the answer is plausible and partly supported, but expected
  page evidence is missing or references are incomplete.
- `ungrounded`: the answer is not supported by returned references.
- `unknown`: references do not expose enough information to judge grounding
  deterministically.
- `judge_error`: the judge could not produce a valid classification.

### Hallucination Risk

Allowed values:

- `low`: no unsupported claims are apparent.
- `medium`: answer includes plausible but not clearly supported details.
- `high`: answer includes unsupported, contradictory, or fabricated claims.
- `unknown`: insufficient information to assess.
- `judge_error`: the judge could not produce a valid classification.

### Overall

Allowed values:

- `pass`: answer is correct or partially correct, grounded or weakly grounded,
  and hallucination risk is low.
- `review`: answer may be useful but needs human inspection.
- `fail`: answer is incorrect, ungrounded, not answered, or high hallucination
  risk.
- `error`: agent or judge failed.

Recommended mapping:

- `pass` when:
  - `answer_correctness` is `correct`; and
  - `grounding` is `grounded` or `weakly_grounded`; and
  - `hallucination_risk` is `low`.
- `review` when:
  - `answer_correctness` is `partially_correct`; or
  - `grounding` is `weakly_grounded` or `unknown`; or
  - `hallucination_risk` is `medium`.
- `fail` when:
  - `answer_correctness` is `incorrect` or `not_answered`; or
  - `grounding` is `ungrounded`; or
  - `hallucination_risk` is `high`.
- `error` when the agent or judge fails.

## Result JSONL Schema

Each result row must include:

```json
{
  "id": "golden_...",
  "question": "string",
  "expected_answer": "string",
  "agent_answer": "string",
  "source_pdf_name": "string",
  "page_number": 1,
  "references": [],
  "reference_check": {
    "expected_pdf_found": true,
    "expected_page_found": true,
    "reference_match_status": "pdf_and_page_found"
  },
  "judge": {
    "answer_correctness": "correct",
    "grounding": "grounded",
    "hallucination_risk": "low",
    "overall": "pass",
    "confidence": 0.9,
    "rationale": "short explanation"
  },
  "agent_error": null,
  "judge_error": null,
  "conversation_id": "string",
  "response_id": "string",
  "usage": {},
  "latency_ms": 1234
}
```

The result schema intentionally uses `expected_answer` to distinguish the golden
answer from the live `agent_answer`, even though the input golden dataset uses
the shorter field name `answer`.

## Summary Report

The runner must write a summary JSON file with:

- Total records.
- Completed records.
- Agent errors.
- Judge errors.
- Counts by `answer_correctness`.
- Counts by `grounding`.
- Counts by `hallucination_risk`.
- Counts by `overall`.
- Counts by `reference_match_status`.
- Pass, review, fail, and error rates.
- Output file paths.

The runner must also print a concise console summary table at the end. The table
must include at least total records, completed records, pass count, review count,
fail count, error count, agent error count, and judge error count.

## Error Handling

The runner must continue by default when individual records fail.

Per-record failures must be captured in the result JSONL with either
`agent_error` or `judge_error`.

The process must return:

- `0` when the run completes and no threshold failure is requested.
- Non-zero when required configuration is missing.
- Non-zero when the dataset is missing, empty, or invalid.
- Non-zero when the result or summary output cannot be written.
- Non-zero when `--fail-on-threshold` is enabled and the configured threshold is
  not met.

## Security

The runner must not log secrets, API keys, bearer tokens, signer internals, or
full environment dumps.

The first implementation targets unsecured evaluation endpoints. Future
versions may add optional bearer-token support.

## Test Expectations

Unit tests must cover:

- Golden dataset loading for the compact schema.
- Agent request payload construction.
- Agent success response parsing.
- Agent HTTP failure handling.
- Agent JSON error handling.
- Reference PDF matching.
- Reference page matching.
- `unknown` page status when references do not expose page information.
- LLM judge prompt construction.
- Judge JSON parsing and validation.
- Overall classification mapping.
- Result JSONL writing.
- Summary aggregation.
- CLI validation errors.

Tests must mock agent HTTP calls and LLM judge calls. Unit tests must not
require live agent endpoints or live OCI model calls.

## Acceptance Criteria

- A specification exists before implementation.
- The runner can read a golden dataset JSONL file.
- The runner can call a configured agent `/responses` endpoint once per
  question.
- The runner records agent answer, references, usage, identifiers, and latency.
- The runner checks whether expected PDF and page evidence appears in
  references.
- The runner uses the evaluation model, not the runtime agent model, as an LLM
  judge.
- The runner classifies correctness, grounding, hallucination risk, and overall
  result.
- The runner writes result JSONL and summary JSON outputs.
- The runner prints a concise totals table to stdout.
- Unit tests cover agent client behavior, reference checks, judge parsing,
  result writing, and summary aggregation.
